# Inline intake confirmation for medication alerts (v2)

The alert component now shows a Yes/No radio group per medication reminder, backed by a per-card `useTransition` to isolate pending state. A new server action `recordMedicationIntakeFromAlert` records intake with a duplicate guard that skips logging when the medication is already marked taken today. The dismiss (X) button is retained for closing alerts without answering.

**Watch for:** `handleConfirm` closes over `selectedAnswers` state, creating a fragile closure pattern that could silently send the wrong answer under future refactoring (likely). Auto-dismiss path bypasses `dismissAlert`, leaving an asymmetric cleanup seam (confirmed).

**Verdict**: APPROVED

## High-level view

All three v1 issues are resolved: per-card `useTransition` via `AlertCard` extraction, duplicate-log guard in the server action, and dismiss button restored.

Two minor concerns remain. `handleConfirm` reads the radio answer from its closure over `selectedAnswers` rather than receiving it as an argument from the child—functional today due to React's synchronous batching, but fragile if the component is later wrapped in a concurrent boundary or the callback reference is memoized differently. Separately, the auto-dismiss timer removes alerts via raw `setAlerts` without calling `dismissAlert`, creating an asymmetric cleanup path. The invariant that the timer only fires when no radio is selected means no orphan key is produced today, but the seam is latent.

## Stale closure in handleConfirm

`handleConfirm` depends on `selectedAnswers` in its `useCallback` deps. The answer lookup happens inside the parent's closure rather than being passed from the child that owns the interaction. A tighter pattern:

```ts
// In AlertCard:
onClick={() => onConfirm(alert, startTransition, currentAnswer)}
```

This removes the closure dependency and makes the data flow explicit.

## Auto-dismiss cleanup asymmetry

Two paths remove alerts:

1. `dismissAlert(alertId)` — removes from `alerts` **and** `selectedAnswers`.
2. The auto-dismiss timer — calls `setAlerts(prev => prev.filter(...))` only.

The timer is cancelled once `selectedAnswers[alert.id] != null`, so path 2 only fires when there's no key to clean. The concern is that this invariant is implicit—a future change that allows the timer to survive interaction would silently introduce orphan state entries.
