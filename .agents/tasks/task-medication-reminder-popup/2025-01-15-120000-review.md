# Inline intake confirmation for medication alerts

The alert component now shows a Yes/No radio group asking whether the user took their medication, replacing the old dismiss-only button. Confirming "Yes" calls a new server action that records the intake to history and marks the medication as taken today; confirming "No" dismisses the alert without persisting anything. Auto-dismiss is suspended once the user begins interacting with the radio buttons.

**Watch for:** Shared `isPending` state across multiple simultaneous alerts blocks interaction on all cards while any single confirmation is in flight (confirmed). The server action unconditionally logs a duplicate entry if the medication was already marked taken earlier in the day (likely).

## High-level view

The component gained per-alert radio selection state and a `useTransition`-backed confirm flow that calls into a new server action. The auto-dismiss timer logic cancels its timeout when a user starts interacting, preventing the alert from vanishing mid-decision—but if the user selects a radio button and never confirms, the alert remains on screen indefinitely.

The new server action `recordMedicationIntakeFromAlert` mirrors the existing `toggleMedication` but omits the toggle-off path. It always sets `last_taken_date` and always creates a log entry when `taken` is true, with no guard against the medication already being marked taken today. A race between two browser tabs or a manual toggle followed by a late alert confirmation would produce duplicate log rows.

A single `useTransition` is shared across all rendered alert cards, so `isPending` becomes `true` for every card when any one card's confirm is in-flight. This disables radio inputs and confirm buttons on all visible alerts simultaneously, degrading the experience when multiple medications fire at the same time slot.

## Shared transition blocks concurrent confirmations

`useTransition` returns a single `isPending` boolean bound to the component instance. When any alert's confirm triggers `startTransition`, the pending flag goes true and every card's radio inputs and button receive `disabled={isPending}`. With two medications due at the same time (a common real-world scenario—morning meds), the user must wait for the first server round-trip to finish before they can interact with the second card.

The fix is either to track pending state per alert ID (a `Set<string>` in state) or to extract the confirm section into a child component that owns its own `useTransition`.

## Duplicate log entry on already-taken medication

`recordMedicationIntakeFromAlert` unconditionally writes:

```ts
updateMedication(medicationId, userId, { last_taken_date: today });
createMedicationLog(userId, medicationId, existing.name, today);
```

Compare with `toggleMedication`, which checks `existing.last_taken_date === today` and toggles off if already taken. The alert action has no such guard. A tab left open from the morning could still have an un-dismissed alert; if the user marks the medication taken via the main UI and then clicks confirm on the stale alert, a second log row is created. The `updateMedication` call is idempotent (sets the same date again), but `createMedicationLog` is not—it appends unconditionally.

## No manual dismiss path

The previous implementation had an explicit dismiss (X) button. The new version removes it entirely—the only way to close an alert is to pick a radio option and confirm, or wait for the 60-second auto-dismiss. This means there's no escape hatch for a user who sees a stale alert for a medication they've already taken via the main checklist and doesn't want to answer the question.
