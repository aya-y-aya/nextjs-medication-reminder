# In-memory Map store replacing better-sqlite3 for Vercel serverless

The data layer is swapped from better-sqlite3 (native C++ addon incompatible with Vercel lambdas) to an in-memory JavaScript store backed by Maps. All 7 API routes, server actions, and the SSR page component now call typed accessor functions exported from `lib/db.ts` instead of writing raw SQL. The `next.config.ts` is left clean (no `serverExternalPackages` referencing the removed dependency). Since the PRD says "no history" and only today's data matters, the ephemeral store is an acceptable MVP trade-off.

**Watch for:** Reminder deduplication is broken by cold starts (confirmed). Non-null assertions on `updateMedication` return value create a crash path if a concurrent delete occurs (confirmed). The `Medication` type makes `last_reminded_at` optional (`?`) but the store always writes it as `null`, creating a subtle type mismatch in the update path (confirmed).

## High-level view

The new `lib/db.ts` exposes a CRUD interface over two Maps (users, medications) initialized lazily via a module-level singleton. Every consumer that previously imported `getDb()` and wrote SQL now imports individual named functions, eliminating the raw SQL surface entirely.

The reminder deduplication logic (`last_reminded_at`) was designed around durable storage. With an in-memory store that resets on cold start, the dedup key is lost and duplicate emails can fire after a Vercel function scale-to-zero cycle. For an MVP demo without real email sending this is cosmetic, but it's the one behavioral regression in the change.

The serverless concurrency model means each warm instance shares the module-level `store` variable, but separate instances (or cold starts) get independent stores. The reminder cron could be routed to a container with an empty medication list and silently do nothing.

<summary>Issues (4)</summary>

1. **Reminder dedup lost on cold start** — `last_reminded_at` resets when the function cold-starts, so the cron job can send duplicate emails within the same time slot. Accept for MVP or gate email sending behind a feature flag.
2. **Non-null assertion crash in PATCH** — `updated!.id` will throw if a concurrent request deletes the medication between the `getMedication` check and the `updateMedication` call. Add a null check or return 404 if `updated` is undefined.
3. **Type mismatch on `last_reminded_at`** — The `Medication` interface marks it as `last_reminded_at?: string | null` (optional), but `createMedication` always sets it to `null`. Pin it to required `string | null` in the type to remove ambiguity.
4. **No persistence across deploys** — Medications added by a user vanish on the next cold start. Acceptable per PRD ("no history"), but worth noting in deployment docs or a UI notice.

## Singleton store lifecycle and serverless implications

The reminder cron (`/api/reminders/check`) could be routed to a different lambda container than the one holding the user's medications, meaning it sees an empty medication list and sends no reminders. This is invisible in local dev (single process) but present in production. For the same reason, if Vercel spins up a second instance under load, one instance's medication additions won't be visible to the other.

## Reminder deduplication regression

The real failure scenario:

1. User adds medication, function stays warm.
2. Cron fires, matches time, sends email, writes `last_reminded_at`.
3. Function goes cold, comes back.
4. Cron fires again for same time slot — `last_reminded_at` is now `null` again — duplicate email.

Since medications themselves aren't persisted across cold starts either, the duplicate scenario only manifests within a single warm lifecycle where the function goes cold and comes back within the same clock minute. In practice, cold start means empty medication list, so no emails fire at all. The real behavioral gap is that the entire reminder feature becomes a no-op after any cold start, not that it sends duplicates.

## Non-null assertions in PATCH handler

```typescript
const updated = updateMedication(medicationId, 1, updates);

return Response.json({
  id: updated!.id,
  user_id: updated!.user_id,
  ...
});
```

`updateMedication` returns `undefined` if the medication doesn't exist or the `userId` doesn't match. The handler checks `getMedication` earlier and returns 404 if missing, but there's no atomicity guarantee — another request could delete the medication between the check and the update. In that case `updated` is `undefined` and the non-null assertions throw a runtime error, returning a 500 to the client. The fix is a simple null guard returning 404.

## Type definition for `last_reminded_at`

The `Medication` interface defines `last_reminded_at?: string | null` — the `?` makes it optional (i.e., the key might not exist on the object). But `createMedication` always sets it to `null`, so in practice the field is always present at runtime. Changing to `last_reminded_at: string | null` (required, non-optional) would align the type with the runtime invariant and let TypeScript catch any constructor that forgets the field.
