# Medication intake confirmation via alert radio buttons

The alert notification popup now includes a "Did you take this medicine?" confirmation section with Yes/No radio buttons and a Confirm button. Selecting Yes calls a new `confirmMedicationIntake` server action that marks the medication as taken and logs to history, then dismisses the alert. Selecting No dismisses without saving. The auto-dismiss timer is paused once the user interacts with a radio button, preventing data loss mid-confirmation.

**Watch for:** Shared `isPending` state across multiple simultaneous alerts can disable all Confirm buttons when only one is saving (confirmed). The server action skips the existence check that sibling actions perform, meaning `updateMedication` silently returns `undefined` for invalid IDs rather than throwing (confirmed, low severity since the DB layer enforces ownership).

## High-level view

The server action mirrors `toggleMedication` but omits the pre-update existence check. `updateMedication` enforces ownership at the DB layer, but when it returns `undefined` (deleted medication), `createMedicationLog` still executes unconditionally, producing an orphan log entry.

The `useTransition` hook is shared across all visible alerts. The single `isPending` boolean disables every Confirm button in the UI when any one alert is saving, not just the one currently in flight.

If the user selects a radio button but never clicks Confirm, the alert persists indefinitely — the auto-dismiss timer is cancelled on interaction and never reinstated.

<details>
<summary>Issues (3)</summary>

1. **Shared `isPending` disables all alerts** — The single `useTransition` is shared across all alert cards. When one alert is saving, every other alert's Confirm button shows "Saving..." and becomes disabled. Use a per-alert pending state (e.g., a `Set<string>` of saving alert IDs) instead.
2. **No existence guard in `confirmMedicationIntake`** — Unlike `toggleMedication`, this action doesn't call `getMedication` before writing. If `updateMedication` returns `undefined` (medication deleted or wrong user), the action still calls `createMedicationLog`, creating an orphan log entry. Add a guard that checks the return value of `updateMedication` and skips the log if it fails.
3. **Double-tap on already-taken medication** — If the medication was already marked taken today (by the checkbox elsewhere in the UI), confirming Yes in the alert will overwrite `last_taken_date` with the same value and create a duplicate log entry. Consider checking `last_taken_date === today` before logging.

</details>

<details>
<summary>Details</summary>

## Shared transition state across concurrent alerts

`useTransition` produces a single `isPending` boolean for the component. The Confirm button's `disabled` prop uses this shared flag:

```tsx
disabled={confirmingAlerts[alert.id] == null || isPending}
```

When two alerts are visible simultaneously and the user confirms one, both Confirm buttons become disabled and show "Saving..." until the transition completes. This creates confusion when multiple reminders fire at the same time (e.g., a user with medications scheduled at the same hour).

## Server action existence gap

The existing `toggleMedication` action fetches the medication first:

```ts
const existing = getMedication(id, userId);
if (!existing) throw new Error("Medication not found");
```

The new `confirmMedicationIntake` skips this:

```ts
updateMedication(medicationId, userId, { last_taken_date: today });
createMedicationLog(userId, medicationId, medicationName, today);
```

`updateMedication` returns `undefined` when the medication doesn't exist or doesn't belong to the user, but the return value isn't checked. The subsequent `createMedicationLog` always executes, which means a deleted medication can still produce a history entry.

## Abandoned alert persistence

If the user selects a radio button and then never clicks Confirm, the alert will never auto-dismiss and never be cleaned up unless the user manually clicks the X button. The timer cancellation fires on any radio interaction and is never reinstated:

```tsx
if (confirmingAlerts[alert.id] !== undefined) {
  if (dismissTimersRef.current.has(alert.id)) {
    clearTimeout(dismissTimersRef.current.get(alert.id));
    dismissTimersRef.current.delete(alert.id);
  }
  continue;
}
```

This means an abandoned alert persists indefinitely until page navigation.

## Duplicate intake logging

If the user confirms intake via the main medication list checkbox *after* an alert has already appeared, the alert remains visible. Confirming Yes in that alert creates a second log entry because `confirmMedicationIntake` doesn't check whether `last_taken_date` already equals today before writing.
