# Client-side medication alert toasts with browser notifications

This change adds a client component that polls the medication list every 30 seconds, compares each medication's `reminder_times` against the current clock minute, and renders toast-style alerts (fixed top-right) when a match occurs. It also fires browser Notification API popups for granted permissions. The component is mounted in the server-rendered `page.tsx` and receives the full medications array as a prop.

**Watch for:** The deduplication key conflates `time` and `currentTime` even though they're always equal at the point of construction (confirmed). The auto-dismiss timeout resets on every render cycle that has active alerts, so alerts can persist longer than 60 seconds if new alerts keep arriving (confirmed). The 30-second poll interval means a reminder can fire up to 29 seconds late or miss entirely if the matching minute falls between two polls (confirmed).

## High-level view

The alert mechanism is purely client-side: it receives a static snapshot of medications from the server at page load and compares reminder times against the browser's local clock. There is no real-time channel and no server push, so the medications list goes stale if the user leaves the tab open across schedule changes.

The deduplication logic in `notifiedRef` relies on a key structure that is redundant (`time` and `currentTime` are identical when the key is created), and the cleanup pass uses a naive `.split("-").slice(-1)[0]` extraction that will break if the key format or ID structure ever changes. This is a latent parsing bug.

The component does not interact with the existing email reminder API route (`/api/reminders/check`). The two notification channels — server-side email via Resend and client-side toasts — operate with no shared state, so there is no mechanism to suppress a client alert if the user already acknowledged the email.

<summary>Issues (4)</summary>

1. **Deduplication key is redundant and cleanup parsing is fragile** — The key `${med.id}-${time}-${currentTime}` always has `time === currentTime`, so the third segment adds nothing. Worse, cleanup splits on `-` and takes the last segment, but `time` is `"HH:MM"` which doesn't contain `-`, so it works today — yet if `med.id` is ever a compound ID with dashes, the split logic breaks. Use a delimiter that can't appear in any segment (e.g., `|`).
2. **30-second poll can miss a one-minute window** — If the component mounts (or an interval fires) at 08:00:28, the next check is at 08:00:58, then 08:01:28. If the reminder is at 08:01, neither check lands in that minute. Reduce the interval to ≤15 seconds or align checks to the start of each minute.
3. **Auto-dismiss timeout resets on each new alert** — The `useEffect` watching `alerts` clears and re-creates the timeout whenever the array changes. If a second alert arrives 50 seconds after the first, the first alert lives 110 seconds total. Use per-alert timeouts or filter by individual timestamps without resetting the timer.
4. **Stale medication data after page load** — Medications are fetched once during SSR and passed as a prop. If the user adds a medication in another tab or the schedule changes, the alert component won't see it until a full page reload. Consider polling an API endpoint or using a client-side fetch to refresh the list.

## Deduplication key structure and cleanup fragility

The deduplication key on line 69 is `${med.id}-${time}-${currentTime}`. At the point where this line executes, `time === currentTime` is guaranteed by the enclosing `if (time === currentTime)` guard on line 68. The key is effectively `${med.id}-${time}-${time}` — the third segment is always a duplicate of the second.

The cleanup pass at lines 94–98 extracts the "time" portion via:

```ts
const keyTime = key.split("-").slice(-1)[0];
```

Because the time format is `HH:MM` (no dashes), this accidentally works: the last segment after splitting on `-` is `currentTime` (e.g., `"08:30"`). But any change to the key format or ID structure could silently break cleanup, causing the `notifiedRef` set to grow unbounded for the session.

## Poll interval and the missed-minute window

The component checks every 30 seconds. A reminder time is a single `HH:MM` string. In the happy case a 30-second interval guarantees at least one check per minute, but if the initial mount or a browser throttle shifts the phase:

```
08:00:59 → check → currentTime = "08:00"
08:01:29 → check → currentTime = "08:01" → MATCH (29s late)
```

In a worst case with a backgrounded tab where `setInterval` is throttled to once per minute by the browser, the 30-second interval becomes effectively 60 seconds and a minute can be skipped entirely.

## Auto-dismiss lifecycle

The auto-dismiss `useEffect` on lines 104–111 depends on `[alerts]`. Every time a new alert is pushed into state, the effect re-runs: it clears the old timeout and sets a fresh 60-second timeout. This means existing alerts get their dismissal clock reset whenever a new alert arrives. For a user with multiple medications at slightly staggered times, early alerts persist well beyond 60 seconds.
