# Cookie-based auth and daily history tracking (v2)

This change adds credential-based JWT session auth (jose + bcryptjs) and a daily history page to the medication reminder app. All routes are protected by middleware, with server actions and API handlers drawing user identity from the session cookie. The v1 review flagged seven issues; five have been addressed — shared secret extracted, `@types/bcryptjs` moved to devDependencies, timezone validation added at registration, history page now uses user timezone, and redundant middleware path checks removed. Two items (rate limiting, N+1 queries) were accepted as MVP limitations.

**Watch for:** Water reset endpoint uses UTC date instead of user timezone, creating a day-boundary mismatch with the rest of the app (confirmed). The well-known fallback JWT secret still deploys silently in production if `AUTH_SECRET` is unset — no startup guard (confirmed).

**Verdict**: APPROVED

## High-level view

One timezone inconsistency slipped through the v1 fixes: `app/api/water/reset/route.ts` computes "today" via `new Date().toISOString().split("T")[0]` (UTC) rather than calling `getUserToday(session.userId)`. Every other write path in the app uses `getUserToday`, so this endpoint will disagree on the current date for users west of UTC after midnight UTC.

The silent fallback secret (`"dev-secret-do-not-use-in-prod"`) remains a fail-open condition for production deployments that omit `AUTH_SECRET`. A startup assertion would prevent accidental exposure without affecting local dev.

## Water reset timezone inconsistency

The water reset endpoint computes the current date differently from every other code path:

```typescript
// app/api/water/reset/route.ts
const today = new Date().toISOString().split("T")[0];
```

Compared to the pattern used everywhere else:

```typescript
// lib/actions.ts, app/api/water/route.ts, app/api/medications/[id]/toggle/route.ts
const today = getUserToday(session.userId);
```

`getUserToday` uses the user's stored timezone to determine the date boundary. The reset endpoint uses UTC. For a user in `America/Los_Angeles` at 10pm local (5am UTC next day), a reset would stamp the wrong date, causing the dashboard's day-boundary logic to see a mismatch and potentially re-reset the counter the next time it renders.

## Fallback JWT secret without production guard

`lib/auth.ts` exports the signing secret with a hardcoded fallback:

```typescript
export const SECRET = new TextEncoder().encode(
  process.env.AUTH_SECRET || "dev-secret-do-not-use-in-prod"
);
```

A deployment that omits `AUTH_SECRET` silently runs with a publicly known key — anyone who reads the source can forge session tokens. A guard like:

```typescript
if (process.env.NODE_ENV === "production" && !process.env.AUTH_SECRET) {
  throw new Error("AUTH_SECRET must be set in production");
}
```

would make this fail-closed without affecting local dev.
