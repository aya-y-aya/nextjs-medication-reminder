# Cookie-based auth and daily history tracking for medication reminder

This change adds two features to the medication reminder app: a credential-based authentication system using JWT session cookies (jose + bcryptjs), and a daily history page that logs medication-taken and water-intake events with timestamps. All existing server actions and API routes are converted from a hardcoded `user_id=1` to session-derived user IDs, with a middleware layer that gates access to all routes except `/login`, `/register`, and the cron-based reminders endpoint. The in-memory store gains user creation, log tables, and history queries.

**Watch for:** Hardcoded JWT secret shared between middleware and auth module with no rotation mechanism (confirmed). Dual auth enforcement in API routes — middleware already blocks unauthenticated requests, but each handler also calls `getSession()` and returns 401, creating a redundant layer that could drift (confirmed). `@types/bcryptjs` placed in `dependencies` rather than `devDependencies` (confirmed). History page displays times in server timezone rather than user timezone (confirmed).

## High-level view

The auth system is a minimal JWT-cookie implementation split across three modules: `lib/auth.ts` (token creation/verification, cookie management), `lib/auth-actions.ts` (login/register/logout server actions), and `middleware.ts` (route gating). The middleware and the auth module duplicate the same secret constant and the same verification logic, which means any future change to the signing algorithm or secret source must be applied in two places.

All API routes received inline session checks (`getSession()` + 401 response) despite the middleware already rejecting unauthenticated requests before they reach the handler. This creates maintenance surface — if one layer is relaxed, the other silently compensates with no visibility, and the two layers return different error response shapes.

The history feature logs events at write time and renders them on a server component that iterates dates calling per-date query functions. The `formatTime` helper uses `toLocaleTimeString` without a timezone argument, so displayed times will reflect the server's locale rather than the user's configured timezone — inconsistent with the date logic in `getUserToday`.

Registration accepts a free-text timezone field with no server-side validation. An invalid timezone string (e.g., `"Foo/Bar"`) passes registration successfully but throws a `RangeError` when the user later visits the dashboard, because `getUserToday` passes it to `toLocaleString`.

<summary>Issues (7)</summary>

1. **Hardcoded shared secret** — `"dev-secret-do-not-use-in-prod"` is duplicated in `middleware.ts` and `lib/auth.ts`. Extract to a shared constant so rotation or environment override happens in one place.
2. **Dual auth enforcement drift risk** — Middleware blocks unauthenticated API requests with 401, and each API handler also independently checks the session. Document which layer is authoritative or remove the redundant handler-level checks.
3. **`@types/bcryptjs` in dependencies** — Type-only packages belong in `devDependencies`; they ship no runtime code and inflate the production install.
4. **No rate limiting on login** — The login action has no throttle or lockout mechanism. An attacker can brute-force credentials at server-action speed. (likely — acceptable for MVP, worth noting for production path)
5. **History N+1 access pattern** — `HistoryPage` calls `getMedicationLogsByDate` and `getWaterLogsByDate` in a loop per date. Acceptable with the in-memory store but will require a batch query if a database replaces it.
6. **Timezone free-text input with no validation** — Registration accepts arbitrary timezone strings. An invalid timezone will throw a `RangeError` on the dashboard via `getUserToday`. Validate with `Intl.DateTimeFormat` at registration time.
7. **History times displayed in server timezone** — `formatTime` in `app/history/page.tsx` calls `toLocaleTimeString` without specifying the user's timezone, so log times render in server-local time rather than user time.

## Shared secret and token verification duplication

The JWT signing secret is constructed identically in two files:

```typescript
// middleware.ts
const SECRET = new TextEncoder().encode(
  process.env.AUTH_SECRET || "dev-secret-do-not-use-in-prod"
);

// lib/auth.ts
const SECRET =
  new TextEncoder().encode(process.env.AUTH_SECRET || "dev-secret-do-not-use-in-prod");
```

Both implement token verification with the same shape check (`typeof payload.userId === "number" && typeof payload.email === "string"`). If the payload schema evolves (e.g., adding roles), both must be updated in lockstep. Extracting a shared module that exports the encoded secret and the verification function would eliminate this.

The fallback `"dev-secret-do-not-use-in-prod"` means a production deployment that forgets the `AUTH_SECRET` env var will silently use a well-known secret. A startup check or build-time validation would catch this.

## Dual auth layer in API routes

The middleware rejects unauthenticated requests to `/api/*` (except `/api/reminders/check`) with a 401 before the handler runs. Each handler then also calls `getSession()` and returns 401 if null:

- The handler-level check can never fire in normal operation because the middleware already blocked the request.
- If middleware is disabled or reconfigured (Next.js 16 already warns `middleware` is deprecated in favor of `proxy`), the handler checks become the sole protection.
- The two layers return different error shapes — middleware uses `NextResponse.json(...)` while handlers use `Response.json(...)`. A client parsing errors might see inconsistent formats depending on which layer fires.

If the intent is that handlers are self-protecting, the middleware's API 401 could be removed so middleware only redirects pages. Either way, documenting the authoritative layer prevents future confusion.

## Registration validation gaps

Registration validates email presence, minimum password length (6 chars), and confirmation match. It does not validate the timezone string server-side — passing `"Foo/Bar"` stores successfully but throws a `RangeError` when `getUserToday` later calls `toLocaleString("en-US", { timeZone: timezone })`. The error surfaces as an unhandled server error on the dashboard rather than a clean registration rejection. A one-line `Intl.DateTimeFormat(undefined, { timeZone })` check at registration time would fail-fast with a user-friendly message.

No rate limiting exists on the login server action. Since this is in-memory and ephemeral the risk is bounded, but the pattern sets a precedent for a production version.

## History page timezone mismatch

The `formatTime` function in `app/history/page.tsx`:

```typescript
function formatTime(isoString: string): string {
  const date = new Date(isoString);
  return date.toLocaleTimeString("en-US", {
    hour: "numeric",
    minute: "2-digit",
    hour12: true,
  });
}
```

This renders in the server's timezone. The user's timezone is available (stored in the user record) and used by `getUserToday` for date boundaries, but the history page doesn't pass it to `toLocaleTimeString`. A medication logged at 11pm Eastern will display as 3am (or 4am) the next day if the server runs in UTC — potentially confusing since it would appear under the wrong date heading.

## Middleware deprecation and redundant path checks

Next.js 16 warns that the `middleware` file convention is deprecated in favor of `proxy`. The code works today but will need migration. The matcher config already excludes `_next/static`, `_next/image`, and metadata files via a negative lookahead regex, yet the function body re-checks `pathname.startsWith("/_next")` and `pathname.includes(".")`. These are dead code paths given the matcher — removing them would reduce noise.
