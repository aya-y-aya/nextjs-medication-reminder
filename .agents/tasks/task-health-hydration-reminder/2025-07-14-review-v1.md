# Health & Hydration Reminder: full-stack MVP

A single-page Next.js app that tracks daily medication completion and water intake, backed by SQLite via better-sqlite3. The implementation delivers a 2-table schema (users, medications), a CRUD API layer for medications and water, an email reminder dispatch endpoint using Resend, and a responsive React dashboard with a medication checklist, water progress bar, and add-medication form. Server Actions handle mutations with `revalidatePath` for SSR freshness.

**Watch for:** Server-timezone date logic applied where user-timezone was specified (confirmed); hardcoded `user_id = 1` eliminates any multi-user path and leaves no authentication surface (confirmed); the reminder endpoint is unauthenticated and has no deduplication, so a repeated call within the same minute fires duplicate emails (confirmed); no test coverage exists anywhere in the diff (confirmed).

## High-level view

The API surface exposes REST-style routes for medications (CRUD + toggle) and water (GET, POST, reset) plus a POST `/api/reminders/check` endpoint designed to be called by an external cron. All route handlers and server actions hardcode `user_id = 1`, so the system is strictly single-user. The toggle, checklist, and water reset all derive "today" from `new Date()` on the server, which returns UTC. The PRD specifies user-localized time for the checklist reset, but only the reminder-check endpoint actually respects the user's timezone. This means the checklist ticks over at midnight UTC rather than the user's local midnight.

The reminder-check endpoint correctly localizes time via `Intl.DateTimeFormat` when matching reminder times, but it does not record that a reminder was sent. A second invocation in the same minute (or a slow cron retry) will send duplicate emails. For the PRD's "99.9% within 60 seconds" delivery metric, the lack of a `last_reminded_at` column or any idempotency key is a gap.

The PATCH endpoint for medications accepts body fields without validation, diverging from the POST endpoint which validates name format and reminder_times array structure. This asymmetry allows malformed data to be written to the database through the update path.

## UTC "today" vs user-local "today"

Throughout the codebase — in `page.tsx`, `lib/actions.ts`, the toggle route, and the water route — "today" is computed as:

```ts
const today = new Date().toISOString().split("T")[0];
```

This returns the UTC date. A user in `America/New_York` at 11 PM local time (03:00 UTC next day) will see their checklist reset prematurely and their water intake zeroed because the server considers it "tomorrow" already. The PRD explicitly stores a timezone on the user record and expects the checklist to "automatically uncheck itself the next day" in the user's local time.

The reminder-check endpoint already demonstrates the correct pattern: using `Intl.DateTimeFormat` with `timeZone: userTimezone` to get the localized time. The same technique should be applied wherever "today" is computed, pulling the user's timezone from the database and formatting accordingly.

## Reminder dispatch without idempotency

The `/api/reminders/check` endpoint scans all users and medications, matches reminder times against the current minute, and sends emails. Nothing records that a reminder was already sent for a given time slot on a given day. The architecture depends on an external cron hitting this endpoint once per minute, but crons can retry on timeout, and nothing prevents a manual invocation from doubling up.

A minimal fix is a `last_reminded_at` or `last_reminded_date` column on medications (or a separate small log table keyed by `medication_id + date + time`) checked before dispatch. The `last_taken_date` column already demonstrates the "overwrite" pattern this schema relies on; a parallel column for reminders would maintain the 2-table constraint if stored on the medications row.

## Unauthenticated reminder endpoint

The reminder-check endpoint is world-accessible — any HTTP client can trigger email sends to all users. Even in an MVP context where the app is single-user with no login flow, the email-dispatching endpoint should be gated by a shared secret (e.g., a `CRON_SECRET` environment variable checked against an `Authorization` header). This endpoint actively costs money (Resend API calls) and delivers unsolicited email if abused.

## PATCH validation gap

The POST handler in `app/api/medications/route.ts` validates `name` (non-empty string) and `reminder_times` (non-empty array of `HH:MM` strings). The PATCH handler in `app/api/medications/[id]/route.ts` accepts these same fields but performs no validation — `body.name` could be an empty string or a number, and `body.reminder_times` could be a non-array value that `JSON.stringify` would happily serialize. Since both endpoints modify the same row, the validation contract should be consistent.

## Test coverage

The diff contains zero test files. The PRD's success criteria (85% adherence rate, 99.9% reminder delivery) imply correctness requirements that manual testing alone won't sustainably verify. At minimum, the reminder-check logic (timezone matching, conditional dispatch) and the day-boundary reset logic deserve test coverage, as both are date-sensitive and easy to break with timezone edge cases.
